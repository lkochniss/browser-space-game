| 2026-06-18 | In-memory models, kein ORM-Mapping bislang | Doctrine installiert aber keine Entity-Attribute. Persistence-Strategie offen — siehe TD-005 |
| 2026-06-18 | Caveman-Tickets (DE/EN mix) | Token-Effizienz, User schreibt DE |
| 2026-06-18 | **Game-Mode = Multi-Player** | Game-Vision (Raumschlacht, Allianz, Handel) impliziert Multi. PvP-Mechaniken zentraler Bestandteil. Tickets T-052/053/054 bleiben relevant |
| 2026-06-18 | **Persistence = Doctrine ORM + Domain-Events** | ORM bereits installiert + Symfony-Standard. Domain-Events via Symfony Messenger als Async-Kanal (Schlachten, Notifications, Score). Volles Event-Sourcing bewusst NICHT initial — kann später ohne Code-Verlust nachgerüstet werden, da Domain-Events bereits etabliert sind |
| 2026-06-18 | **Event-Modell = B (Hybrid: ORM + Domain-Events)** | 90% der Event-Vorteile (Async, Decoupling, Scaling) ohne Wochen Plumbing. Read-Models in ORM, State-Mutations dispatchen Events; Subscriber/Handler über Messenger asynchron |
| 2026-06-18 | Tests via In-Memory SQLite | Standard-Approach. Reicht für Integration-Tests. Prod = MySQL (composer.json `doctrine/dbal:^3`) |
| 2026-06-18 | Aggregate-Inverse-Setter-Pattern | Building/Resource/Deposit haben nullable `?Planet $planet`-Setter; `Planet::addX()` setzt Inverse. Pragmatisch, weil Factories an mehreren Stellen ohne Planet-Ref entstehen. Alternative (Planet-Ref im Konstruktor) wäre breaking + factory-spread. |
| 2026-06-18 | Initial-Migration handgeschrieben (Schema-API) | `doctrine:migrations:diff` braucht Live-DB. Ohne MySQL-Container nutzen wir `Schema`-Builder-API (plattform-agnostisch) für `Version20260618000001`. Folge-Migrations können wieder per `diff` generiert werden, sobald MySQL läuft. |
| 2026-06-22 | **Storage-Volume in m³ (Generic-Storage Foundation T-180)** | Eine physikalische Einheit pro Storage-Bucket statt Resource-Type-spezifischer Caps. Player-intuitiv (1 m³ = bekannte Welt-Einheit). Vision T-177/T-178/T-179: Planet hat einen generischen Volume-Bucket, jede Resource + Pop nimmt m³ proportional zum `ResourceVolumeConfig`-Multiplier. Working-Pop = 10 m³/Person, Wasser = 1 m³ als Reference. T-061 Resource-Category-Caps werden in T-177ff superseded. |
| 2026-06-22 | **Stations sind Lost-Tech (T-174 Soft-Deprecate)** | Station-Bau-Technologie ist im Universum verschollen (40k-Style). `BuildSpaceStationCommand` wirft `StationConstructionDeprecatedException`; Command/Handler/Service-Stub bleibt bis T-175 (Galaxy-Spawn) deployt ist, danach Hard-Remove. Stations können nur via Claim (T-023b ABANDONED) oder Combat-Capture (T-176) übernommen werden. 6 obsolete POI-Exceptions entfernt (Missing/Insufficient/Already-Exists/PlayerNotFound/SolarSystemNotFound), da Validation-Path nicht mehr läuft. |
| 2026-06-22 | **Multi-Lab Opt-In ersetzt Auto-Aggregator (T-025c)** | T-025b Auto-Aggregator (alle Labs stacken kostenlos) wird gelöscht. Stattdessen wählt Player pro Forschung Primary-Lab + N Booster-Labs. Bonus via Geometric-Decay 0.5 (D1), Cost via Flat-Aufschlag 10% pro Booster + quadratische Mismatch-Penalty 2% pro (primary−booster)² (D2). Frozen-at-Start (D4): Lab-Upgrade/-Verlust während laufender Forschung wird ignoriert. JSON-Persistence von Primary + Booster-Planet-IDs auf `active_research`. Default-Verhalten ohne Primary-Argument: Auto-Pick stärkstes Lab, kein Booster (Test-/API-Friction minimal). |
| 2026-06-22 | **Planet-Cap pro Player (T-101)** | Anti-Steamroller: jeder Player hat ein max Planet-Cap. `BASE_CAP=5`, `HARD_CAP=10`; +1 pro `logistics_1`-Level (max 3 mit existing Node, weiter durch T-136 Logistics-Branch). `ColonizePlanetCommandService` validiert vor Pop-Transfer und wirft `PlanetCapReachedException`. Heimat-Planet wird normal mitgezählt (kein Sonder-Slot — T-081 Heimat-Schutz ist separate Mechanik). Abandon-Mechanik ist in T-101b ausgelagert (Open Questions Buildings/Cooldown/Heimat-Schutz). |
| 2026-06-22 | **PropulsionType pro Ship (T-026c)** | Jedes Schiff hat einen Antriebs-Typ (HYDROGEN..JUMPDRIVE, 7 Werte). HYDROGEN ist Foundation-Default (keine Research). Andere brauchen die entsprechende T-026 Antriebs-Forschung beim Build (`BuildShipCommandService` validiert; wirft `PropulsionResearchNotMetException`). `Ship::getEffectiveSpeed()` = `ShipType × PropulsionType.getSpeedMultiplier()`; `Fleet::getMinSpeed()` nutzt das. Inter-System-Travel-Gate läuft weiterhin über Player-Forschung `ftl_hyperdrive` in `MoveFleetCommandService` (Per-Move-Range-Enforcement via `getMaxSystemRange` ist informativ, kommt mit T-026d). |
| 2026-06-22 | **Lab-Tier-Gating (T-069)** | `ResearchNode::requiredLabLevel` (default 1) — Player muss `effectiveLab >= requiredLabLevel` haben um Node zu starten. Mapping: Tier-1 = L1 (alle T-170 Foundation + propulsion_hydrogen), Tier-2 = L2 (propulsion_ion/fusion, ftl_hyperdrive), Tier-3 = L3 (propulsion_antimatter, ftl_warp/jumpdrive). Effective-Lab via T-025c Multi-Lab-Aggregat → Player kann Booster nutzen statt L3-Lab zu bauen (strategischer Tradeoff via T-025c Cost). `LabLevelTooLowException` mit klarer Hint. Power-Hook ist Stub-NoOp (T-065 Draft). |
| 2026-06-22 | **Bubble-Foundation (T-150)** | Newbie-Schutz-Foundation: `Player.bubbleStatus` (BUBBLE/EXITED, default BUBBLE). Auto-Exit beim 2. Planet-Claim in `ColonizePlanetCommandService`. Foundation-Flag ohne aktive Effekte — Skip-Effects (Pirate/Auction/Map), Notifications und Catch-Up-Mining-Multiplier sind in T-150b ausgelagert (warten auf T-074/T-075/T-111/T-160/T-161). Backfill-Migration setzt bestehende Player mit ≥ 2 Planeten direkt auf EXITED. |
| 2026-06-22 | **Heimat-Schutz-Foundation (T-081)** | `Planet.isHomePlanet` Flag (default false). Start-Planet wird automatisch in `ClaimStartPlanetCommandService` via `markAsHome()` markiert. Foundation-only — alle Anti-Crush-Effekte (Pop-Loss-Cap 10%, Resource-Vault 30%, Building-Damage-Cap, Shield-Cooldown 24h, Sensor-Warning) sind in T-081b ausgelagert (warten auf T-103/T-080/T-068/T-161). Per-Player-Uniqueness wird auf Application-Layer durchgesetzt (kein DB-Unique-Constraint, da "Heimat verlegen" Q1 in T-081b offen). Backfill-Migration markiert pro Player den ersten Planeten (`MIN(id)`) als Heimat. T-101b Abandon-Mechanik liest den Flag für Heimat-Block. |
| 2026-06-22 | **Pop-QoL-Buildings Foundation (T-070)** | 4 neue Buildings HOSPITAL/UNIVERSITY/CULTURAL_CENTER/TEMPLE (alle strikt-unique pro Planet, T-171-konform). HOSPITAL +20 Pop-Cap/Lvl (via existing `getPopulationCapBonusPerLevel`-Hook). CULTURAL_CENTER +2%/Lvl auf Mining + Refinement, capped +20% (neuer `Planet::getCulturalCenterMultiplier()` der in `getEffectiveMiningMultiplier` + `getEffectiveRefinementMultiplier` reinmultipliziert). UNIVERSITY + TEMPLE bauen ohne Effekt im Foundation. Mangel-Tod-Reduktion (Hospital), RP-Multi (University), Loyalty (Temple) und Power-Consumption sind in T-070b ausgelagert (warten auf T-005-Hook / Research-Hook / T-122-Verzahnung / T-065). _UNIVERSITY-Anteil revoked via T-182._ |
| 2026-06-23 | **UNIVERSITY-Building entfernt (T-182)** | T-070 hatte UNIVERSITY als eigenes strikt-unique Building eingeführt (Slot-2, +5% RP-Output/Lvl in T-070b geplant). User-Klarstellung: UNIVERSITY == LAB — `RESEARCH_LAB` (T-025) ist die einzige Forschungs-Einrichtung. UNIVERSITY war Wort-Mix-Up und wird ersatzlos gestrichen: Enum-Case + Cost + Duration + Slot-Size + Test-Cases + Doc-Einträge cleanuped. T-070b University-Section + Q1 (RP-Multi-Modell) obsolet. T-089 Pop-Tier-Referenz auf RESEARCH_LAB umgehängt. Demo-State braucht keine Migration — Reset cleart alles. |
